# Requirements

## 1. プロジェクトの目的

`morning-brief` は2つの目的を持つ。両方を達成することがゴールであり、片方だけでは成功とみなさない。

### 1.1 第一の目的: 通勤時の英語ニュースレター消化

ユーザーは Gmail に届く複数の英語のテック系ニュースレターを購読しているが、英語が苦手なため十分に読めていない。「日本語でぱっと読んで、気になる記事だけ原文を読む」という消化スタイルを成立させる仕組みを作る。

### 1.2 第二の目的: ambient agent の学習

ユーザーは将来的に ambient agent 関連の開発に携わる予定がある。実用システムを自分で組むことで、フレームワーク選定・永続化・サーバーレス文脈での運用・HITL 設計・自動学習ループといった要素を素振りする。

## 2. ユーザーストーリー

### US-1: 朝のダイジェスト受信

> ユーザーとして、平日の通勤時に、前日に届いた英語ニュースレターの日本語要約を Slack で受け取り、片手で読み流せる状態でありたい。

### US-2: 興味のある記事への深掘り

> ユーザーとして、要約を読んで気になった記事の原文に1タップで遷移したい。

### US-3: 軽量フィードバック

> ユーザーとして、「この記事よかった」「この送信元はもう要らない」「もっと深く知りたい」を、文章を書かずにリアクション/ボタン1タップで伝えたい。

### US-4: エージェントの成長

> ユーザーとして、フィードバックを重ねるほどダイジェストの内容が自分の興味に最適化されていくのを観察したい。

### US-5: 拡張可能性

> ユーザーとして、将来 Slack 以外の配信先 (Telegram など) や別ソース (RSS など) を追加したい場合に、コア部分を書き換えずに拡張したい。

## 3. 機能要件 (FR)

### FR-1: ニュースレター取得

- 平日 06:30 JST に、Gmail から前24時間以内に届いた未処理メールを取得する。
- 対象は Gmail ラベル `Newsletter/Tech` が付いたものに限定する。ラベル付与は Gmail フィルタ機能で事前に設定する (システムの責務外)。
- 「未処理」の定義: 既存ラベル `Newsletter/Tech/Processed` が付いていないこと。処理完了時にこのラベルを付与する。

### FR-2: 日本語要約

- 各メールについて、Gemini 2.5 Flash で日本語に要約する。
- 要約のフォーマットは2段構え:
  - **TL;DR セクション**: 全メールから上位 3-5 本を抽出。各本につき 2 文程度 (何の話 / なぜ気になるか) と原文 URL。
  - **詳細セクション**: 1メール1ブロック。要点 3 つ + 専門用語注 + 原文 URL。
- 原文 URL は本文中の主要記事のリンクを抽出する。複数ある場合は最も主要なもの 1 つ。

### FR-3: Slack 配信

- 個人用 Slack workspace の `#newsletter-digest` channel に投稿する。
- Block Kit でフォーマットし、各記事ブロックにリアクション促しを付ける:
  - 👍 (もっとこういうの), 👎 (要らない), 🔥 (深掘り希望), 🔇 (送信元ミュート希望)
- `[ミュート]` ボタンは廃止済み (ADR-012)。ミュート意思は 🔇 リアクションで表明する
- 件名相当のヘッダーに日付 (`YYYY-MM-DD`) を含める。

### FR-4: HITL フィードバック収集

- 翌朝の実行時、前日のダイジェスト投稿に対する以下を収集する:
  - 各記事ブロックへの絵文字リアクション (👍/👎/🔥/🔇 等)
  - 🔇 リアクション (送信元ミュート意思表示)
  - スレッド返信 (自由記述)
- 収集したフィードバックは `feedback.jsonl` に追記し、Phase 5 の USER.md 更新サブルーチンで利用する。

### FR-5: 学習・成長

- 収集した feedback.jsonl を Gemini (PydanticAI 経由) に渡し、`seeds/USER.md` と `seeds/MEMORY.md` の差分を生成する (ADR-012 / ADR-013 / ADR-014)。
- 生成した差分は GitHub PR として作成し、ユーザーがマージした時点で確定する。
- フィードバックに応じて以下が更新されることを期待する:
  - `seeds/USER.md`: 好むトピック / 嫌うトピックの記述
  - `seeds/MEMORY.md`: 長期記憶
- 更新履歴は Git log で追跡できる。誤った更新は `git revert` で戻せる。

### FR-6: ドライラン

- `--dry-run` モードで、実際の Slack 投稿・Gmail ラベル更新を行わずに最終 Markdown を stdout に出すことができる。

### FR-7: 失敗時の通知

- 各 Phase が例外で死んだ場合、`#alerts` channel (digest channel とは別) に通知する。
- Phase 2 が空 (対象メールなし) の場合は失敗扱いせず、「本日は対象メールなし」を `#newsletter-digest` に投稿する。

## 4. 非機能要件 (NFR)

### NFR-1: コスト

- 月額コスト目標: ¥500 以下 (Modal 無料枠 + Gemini Flash の従量課金分)。
- Modal の無料枠 ($30/月相当) で運用が完結することを目指す。

### NFR-2: 信頼性

- 平日朝 06:30 JST にダイジェストが届かないことを最も避ける。
- 一部 Phase の失敗で全体が落ちないよう、Phase 間で疎結合に設計する。
- 重要な失敗は `#alerts` で気付ける。

### NFR-3: シングルユーザー前提

- マルチテナント設計は不要。OAuth はユーザー個人の Gmail 1 つに紐付く前提。

### NFR-4: 秘匿情報の分離

- リポジトリは GitHub にオープンソースとして公開する前提。
- 秘匿情報 (API キー、OAuth refresh token、bot token) はリポジトリに含めない。
- リポジトリにはテンプレ (`*.example.yaml`, `.env.example`) のみ。

### NFR-5: 拡張性

- 配信先の追加は `Notifier` Protocol の新実装追加だけで済むこと。
- ソースの追加 (RSS など) も同様に、Phase 2 のソース実装を差し替えれば済むこと。
- 拡張時にコア (`modal_app.py`, 各 Phase の制御フロー) を書き換える必要がないこと。

### NFR-6: 可観測性

- Sprint 2 以降、USER.md の学習が進んでいるかを定量的に観察できること。
- `git log seeds/USER.md` で更新履歴を確認できる。
- フィードバック量・LLM コストを週次でレポートできる仕組みを持つ (Sprint 3 で `scripts/weekly_report.py` を実装)。

### NFR-7: 再現性 (OSS)

- リポジトリを clone して `docs/setup.md` の手順に従えば、別環境で同等のシステムが立ち上がること。
- ただし参照実装としての性格が強く、汎用的実用性は最優先ではない (オーナー 1 名のセットアップ前提でいい)。

## 5. 非ゴール (やらないこと)

明示的にスコープ外とする。誘惑があっても今は手を出さない。

- **NG-1**: 完全自動化。フィードバック不要なくらい賢いシステムを目指さない。HITL のループを回すこと自体が学習目的の一部。
- **NG-2**: 他人のためのプロダクト化。マルチユーザー、課金、UI、認証フローの汎用化はしない。
- **NG-3**: 元メールの全文翻訳。要約のみ。深く読みたければ原文 URL に飛ぶ。
- **NG-4**: 即時配信。届くたびに処理ではなく、1 日 1 バッチ。
- **NG-5**: 週末の配信。週末分は月曜の通勤時にまとめて読む。
- **NG-6**: モバイルアプリ開発。Slack の既存アプリを使う。
- **NG-7**: 完全自律エージェント化。Python コードのオーケストレーションを agent loop に置き換えることは現時点では行わない (境界B 維持 / ADR-012)。
- **NG-8**: 厳密な SLA。朝 06:30 に絶対届くことは保証しない。届かない日があっても許容。

## 6. 成功基準

### Sprint 1 完了基準

- 平日朝 06:30 JST に、`#newsletter-digest` にダイジェストが自動投稿される。
- ダイジェストは TL;DR + 詳細の 2 段構え。
- 各記事ブロックにリアクション/ボタンが付いている。
- リポジトリは秘匿情報を含まない状態で push できる。
- `just check` がグリーン。

### Sprint 2 完了基準

- T2.1 〜 T2.4 がマージされている (T2.5 PR 自動化は別タスク)。
- Phase 1 で feedback.jsonl にフィードバックが記録されている。
- Phase 5 で `update_if_ready()` が動き、十分なフィードバックが蓄積したら Gemini が USER.md diff を生成する。
- `just dry-run` で Phase 1 / Phase 5 のログが出る。
- `just check` がグリーン。

### Sprint 3 完了基準

- 配信先の追加 (Telegram など) または別ソース追加 (RSS など) が、コア部分の変更なく完了している。
- これにより `Notifier` 抽象 / Phase 2 のソース抽象が機能したことが実証される。

## 7. ステークホルダー

- **オーナー / ユーザー**: 個人開発者 1 名
- **想定読み手** (OSS): ambient agent / Modal / PydanticAI に関心のある開発者。参照実装として読む層。
